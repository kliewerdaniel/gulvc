# 26 — Local Development

> The on-ramp for a new contributor. From a fresh clone to a running platform in under 30 minutes.

---

## Purpose

This document describes how to set up Vibe for local development. It is the answer to "I just joined, what do I do?" The goal is a reproducible, fast feedback loop that any engineer can stand up in well under 30 minutes.

The platform is a system of services. A local environment must reflect that faithfully enough for development and testing, without the cost of a real production deployment.

## Principles

1. **One command to bootstrap.** New contributors run one command and get a working stack.
2. **Deterministic by default.** Lockfiles are committed; tooling versions are pinned.
3. **Realistic but isolated.** Local services mirror production where it matters; isolation where it doesn't.
4. **Idempotent re-runs.** Bootstrap and tear-down can run repeatedly.
5. **Cheap.** Local development should not require paid SaaS accounts.

## Repository Bootstrap

```bash
git clone git@github.com:vibe-dev/gouplevelvibecoding.git
cd gouplevelvibecoding
make bootstrap
```

`make bootstrap` performs:

1. Verifies tooling versions.
2. Installs pre-commit hooks.
3. Installs pnpm dependencies.
4. Installs Python dependencies via `uv`.
5. Builds workspace outputs (`turbo run build`).
6. Starts the local stack (`docker compose up -d`).
7. Runs database migrations.
8. Seeds minimal test data.
9. Prints a success summary with the local URLs.

A successful bootstrap prints:

```
vibe local stack ready
  API:        http://localhost:8080
  Web:        http://localhost:3000
  Temporal UI: http://localhost:8233
  Postgres:    localhost:5432 (user=vibe, pw=vibe)
  Redis:       localhost:6379
  MinIO:       http://localhost:9001 (console)
  MailHog:     http://localhost:8025
  Traefik:     http://localhost:81
```

## Local Services (docker compose)

`docker compose.yml` defines a self-contained local environment:

| Service | Image | Purpose |
|---------|-------|---------|
| postgres | postgres:16 | primary database |
| redis | redis:7.4 | queue, cache, rate limit |
| minio | minio/minio | S3-compatible storage |
| temporal | temporalio/auto-setup | orchestrator |
| mailhog | mailhog/mailhog | outbound email capture |
| traefik | traefik:v3 | reverse proxy for local dev |
| llm-mock | local image | mocks LLM providers with recorded responses |
| git-mock | local image | mocks GitHub for tests |
| vercel-mock | local image | mocks Vercel for tests |
| stripe-mock | stripe/stripe-mock | mocks Stripe API |

Local secrets are written to `.env.local` and excluded from version control.

## Environment Configuration

`.env.example` lists every variable. `.env.local` is a developer's private copy.

Required variables for local dev:

- `DATABASE_URL` — Postgres connection string
- `REDIS_URL` — Redis connection string
- `S3_ENDPOINT` — MinIO endpoint
- `S3_BUCKET` — `vibe-dev`
- `JWT_SIGNING_KEY` — local-only key, 32+ bytes
- `STRIPE_SECRET_KEY` — `sk_test_...` from a developer test account
- `STRIPE_WEBHOOK_SECRET` — generated by `stripe listen`
- `LLM_API_KEY` — optional; falls back to mocks if absent
- `GITHUB_APP_ID`, `GITHUB_APP_PRIVATE_KEY_PATH` — local fixture
- `VERCEL_TOKEN` — local fixture
- `TEMPORAL_ADDRESS` — `localhost:7233`

`make env` generates `.env.local` from a template with safe defaults.

## Database Migrations

- Migrations are run as part of `make bootstrap`.
- A new migration: `make migration name="add_widgets_table"`.
- The migration is generated via Alembic.
- Migrations are forward-only.
- Locally, you can reset the database with `make db-reset` (destructive; never in production).

## Running the Platform

```bash
make up
```

`make up`:

1. Starts docker compose services.
2. Runs migrations if needed.
3. Starts all apps under a process supervisor (overmind or honcho).

Logs stream interleaved to the terminal. To run a single app:

```bash
make run app=api
```

To run workers:

```bash
make run app=capture-worker
```

## The `vibe` CLI

`vibe` is an in-repo CLI for common tasks. It is built from `tools/cli/`.

```bash
vibe --help
vibe jobs run --url https://example.com
vibe jobs list
vibe agents replay --agent capture --fixture basic
vibe db seed
vibe docs serve
vibe env doctor
vibe token mint --tenant tnt_xxx --role admin
vibe webhook trigger --event job.completed
```

`vibe env doctor` runs a comprehensive check: tooling versions, ports, secrets, and database connectivity.

## Debugging

### Python

- `debugpy` is preconfigured; attach VS Code to port 5678.
- `make debug app=api` starts the app with the debugpy listener.

### TypeScript

- VS Code's Node debugger attaches to a process started with `--inspect=9229`.
- `make debug app=web` exposes port 9229.

### Workflows

- The Temporal UI shows all workflow state.
- `vibe workflow trace <id>` prints a hierarchical view.

### Tracing

- A local Tempo is exposed at `http://localhost:3200`.
- `vibe job trace <id>` opens the trace.

### Logs

- Logs are JSON; pipe to `jq` for readability.
- `make logs svc=api | jq` is a common pattern.

## Testing Locally

```bash
make test                # all
make test pkg=apps/api   # package
make test name=capture   # substring match
make e2e                 # end-to-end job
make perf                # performance smoke
```

Tests are hermetic: they use a separate database, mocks, and recorded fixtures.

## Troubleshooting

### Port conflicts

- `make doctor` lists ports in use.
- Reassign via `.env.local`: `API_PORT=8090`.

### Database connection refused

- `make docker ps` to confirm Postgres is up.
- `make db-reset` to recreate.

### Migration mismatch

- `make db-current` to see state.
- `make db-upgrade` to apply.

### Stuck worker

- `make worker-restart worker=capture`.

### Out of disk

- `make docker-prune` removes unused images.
- MinIO volume is at `~/.vibe/minio`; clear with `make minio-reset` (destructive).

### LLM not responding

- If `LLM_API_KEY` is unset, mocks are used.
- To test real providers, set the key and `LLM_MODE=record` to record real responses.

### Temporal workflow stuck

- `vibe workflow describe <id>` shows state.
- `vibe workflow cancel <id>` cancels safely.
- `vibe workflow reset <id>` to a prior point (advanced).

## Design Decisions

| Decision | Rationale |
|----------|-----------|
| `make` as the entry point | Cross-language, no new tool to learn. |
| Docker Compose for local services | Realistic without infrastructure cost. |
| Recorded LLM fixtures by default | Determinism, no API cost. |
| `vibe` CLI for common tasks | One tool, well-named. |
| Strict pre-commit | Catch issues before they enter CI. |
