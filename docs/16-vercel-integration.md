# 16 — Vercel Integration

> The contract between the platform and Vercel. Tokens, project lifecycle, deployments, domains.

---

## Purpose

This document specifies how Vibe integrates with Vercel. Vercel is the deployment platform for **both** the Vibe frontend and the customer-delivered websites.

This document focuses on the Vercel-specific concerns: token model, project creation, deployment triggers, build verification, domain management, and per-tenant Vercel team linking.

It complements `14-deployment-engine.md` (engine-level pipeline) and `15-github-integration.md` (the upstream Git host).

## Why Vercel

See ADR-004 for the decision record. Summary:

- Best-in-class Next.js hosting.
- First-class build and deploy APIs.
- Automatic image, font, and edge optimizations.
- Pricing model compatible with our COGS targets.
- Custom domain provisioning is straightforward.

## Vercel Account Topology

### Vibe-Owned Team

- A Vercel team named `vibe-platform`.
- Hosts customer sites where the customer has not connected their own Vercel team.
- Limits per Vercel pricing tier are monitored; the platform tracks utilization in its dashboards.

### Tenant-Linked Teams (V2+)

- Customers (typically agency tier) may connect their own Vercel team via OAuth.
- The platform stores the resulting access token in Secrets Manager and a record in `vercel_links`.
- All subsequent deployments for that tenant land in their team.

The default is the Vibe-owned team. Tenant linking is opt-in.

---

## Token Model

### Personal-Like Token (Vibe Team)

- A Vercel "Access Token" scoped to the Vibe team.
- Created once per environment (dev, staging, prod) and stored in Secrets Manager.
- Rotated quarterly; rotation runbook in `27-production-deployment.md`.

### Customer-Linked Token

- Obtained via the Vercel OAuth flow.
- Stored encrypted in Secrets Manager, ARN recorded in `vercel_links.access_token_secret_arn`.
- Refreshed on `401` per the Vercel OAuth protocol.
- Revocable by the customer (or by Vibe on tenant deletion).

## Project Lifecycle

### Create Project

```python
async def create_vercel_project(
    *,
    team_id: str | None,
    token: str,
    name: str,
    git_repo: GitRepo,
    framework: str = "nextjs",
    root_directory: str | None = None,
    env_vars: dict[str, str] | None = None,
) -> VercelProject:
    payload = {
        "name": name,
        "framework": framework,
        "gitRepository": {
            "type": "github",
            "repo": f"{git_repo.owner}/{git_repo.name}",
        },
        "buildCommand": None,         # default to framework preset
        "outputDirectory": None,
        "rootDirectory": root_directory,
        "environmentVariables": [
            {"key": k, "value": v, "target": ["production", "preview", "development"]}
            for k, v in (env_vars or {}).items()
        ],
    }
    url = f"https://api.vercel.com/v10/projects"
    if team_id:
        url += f"?teamId={team_id}"
    async with httpx.AsyncClient() as client:
        resp = await client.post(url, headers={"Authorization": f"Bearer {token}"}, json=payload)
        resp.raise_for_status()
    return VercelProject.parse_obj(resp.json())
```

Project naming:

- Mirrors the GitHub repo name.
- Lowercased, hyphen-separated, max 100 chars (Vercel limit).

### Configure Environment Variables

Baseline env vars set by the platform:

| Key | Value | Purpose |
|-----|-------|---------|
| `NEXT_PUBLIC_SITE_URL` | The Vercel preview URL (replaced on custom domain attach) | Used by sitemap / canonical generation |
| `VIBE_GENERATED` | `"1"` | Indicates the app was generated by Vibe (optional, for diagnostics) |
| `VIBE_JOB_ID` | The originating job UUID | For correlating deployments back to jobs in support tickets (Pro-tier hides this) |

Customer-specific env vars (V2+) are added via the dashboard or job spec.

### Trigger Deployment

GitHub-linked projects deploy automatically on push, but the engine also explicitly creates a deployment to capture the deployment ID:

```python
async def create_deployment(
    *, team_id: str | None, token: str, project_id: str, git_branch: str = "main"
) -> Deployment:
    url = "https://api.vercel.com/v13/deployments"
    if team_id:
        url += f"?teamId={team_id}"
    payload = {
        "name": project_id,
        "project": project_id,
        "target": "production",
        "gitSource": {"type": "github", "ref": git_branch},
    }
    ...
```

### Poll Build Status

```python
async def wait_for_deployment(token: str, team_id: str | None, deployment_id: str) -> Deployment:
    deadline = monotonic() + 900  # 15 minutes
    while monotonic() < deadline:
        d = await get_deployment(token, team_id, deployment_id)
        if d.readyState in ("READY", "ERROR", "CANCELED"):
            return d
        await asyncio.sleep(5)
    raise DeploymentTimeout(deployment_id)
```

`readyState` values handled: `INITIALIZING`, `BUILDING`, `READY`, `ERROR`, `CANCELED`.

### Stream Build Logs

```python
async def stream_build_logs(token: str, team_id: str | None, deployment_id: str) -> AsyncIterator[str]:
    url = f"https://api.vercel.com/v3/deployments/{deployment_id}/events"
    if team_id:
        url += f"?teamId={team_id}"
    async with httpx.AsyncClient(timeout=None) as client:
        async with client.stream("GET", url, headers={"Authorization": f"Bearer {token}"}) as r:
            async for line in r.aiter_lines():
                yield line
```

Streamed logs are persisted to S3 under `build_logs/{deployment_id}.log` and referenced by `deployments.build_log_uri`.

### Custom Domains

```python
async def attach_domain(token: str, team_id: str | None, project_id: str, domain: str) -> DomainInfo:
    url = f"https://api.vercel.com/v10/projects/{project_id}/domains"
    if team_id:
        url += f"?teamId={team_id}"
    resp = await httpx.AsyncClient().post(
        url,
        headers={"Authorization": f"Bearer {token}"},
        json={"name": domain},
    )
    resp.raise_for_status()
    return DomainInfo.parse_obj(resp.json())
```

DNS validation:

- Vercel returns the required CNAME / TXT records.
- The platform surfaces these to the customer via the dashboard and email.
- The platform polls `GET /v9/projects/{id}/domains/{domain}` every 60s for `verified=true`.
- Timeout: 24 hours. After timeout, marked `pending`.

## Failure Mode Matrix

| Failure | Detection | Recovery |
|---------|-----------|----------|
| Token expired (customer-linked) | 401 | Refresh OAuth; if refresh fails, mark `vercel_links` revoked, notify customer |
| Project name conflict | 409 / 422 | Append disambiguator, retry once |
| Build error | poll returns ERROR | Capture logs, route to Generation Agent for analysis, redeploy up to twice |
| Build timeout | 15 min timer | Cancel deploy, compensate |
| Custom domain validation timeout | 24 h | Mark `pending`, deliver default URL |
| Rate limit | 429 | Backoff with jitter |
| API outage | 5xx | Retry, then operator alert |

## Quota Management

Vercel limits vary by plan. The platform monitors:

- Deployments per day (per team)
- Build minutes
- Bandwidth (delivered sites)
- Edge function invocations

A daily rollup job computes per-team utilization and emits alerts at 60%, 80%, 95%.

## Idempotency

Vercel does not natively support idempotency keys. The platform handles this:

- Project creation: check for existing project by name before creating.
- Deployment creation: tag with `meta.vibe_job_id` and check for an existing deployment for the same job before creating.

## Environment Strategy

The platform itself uses Vercel for:

- `vibe-marketing` (the marketing site at `vibe.dev`).
- `vibe-dashboard` (the customer dashboard at `app.vibe.dev`).
- `vibe-admin` (the operator console at `admin.vibe.dev`, IP-restricted).

Each lives in its own Vercel project under the Vibe team. Production branches are protected; previews are auto-deployed for every PR.

Customer-delivered sites are separate projects, named per `{hostname}-vibe-{shortid}`.

## Security Considerations

- Tokens never logged in plaintext.
- Tokens stored encrypted; access governed by IAM.
- Customer-linked tokens scoped to least-privilege Vercel OAuth scopes.
- Generated app does not embed any Vibe-owned credentials.
- See `17-security-model.md`.

## Operational Procedures

### Rotating the Vibe Team Access Token

1. Generate a new token in Vercel.
2. Store in Secrets Manager (new version).
3. Update worker config to point to the new ARN.
4. Roll workers.
5. Revoke the old token after 24 hours.

### Customer Revokes Linked Vercel Team

1. Vercel does not currently push revocation webhooks; detected on `401` on next use.
2. Mark `vercel_links.revoked_at`.
3. Future jobs either route to the Vibe team (if tenant allows) or fail with `vercel_link_required`.
4. Notify customer.

### Vercel Outage

1. Operator triggers "deploy hold" via admin console.
2. New jobs queue but do not call Vercel API.
3. Monitoring detects Vercel recovery; deploy hold released.
4. Queued jobs drain.

## Testing Strategy

- **Unit:** name slugification, env var serialization, idempotency check.
- **Integration:** against a dedicated test Vercel team. Projects and deployments deleted at teardown.
- **Contract:** record-replay against Vercel API.

## Design Decisions

| Decision | Rationale |
|----------|-----------|
| Single Vercel provider | Depth over breadth. |
| Vibe-owned team as default | Removes onboarding friction. |
| Customer-linked teams as opt-in (V2) | Required for white-label. |
| Streaming build logs | Live debug, smaller storage spikes. |
| Custom domain non-blocking | Avoids holding delivery hostage to DNS. |
